## 1. 整体架构风格

系统明确**部署于本地局域网（LAN）内**，采用**前后端分离**架构，Nginx 与 Node.js 分工明确（见第 9 节）：

```
浏览器
  |
  | HTTPS
  v
Nginx（反向代理 + 静态文件）
  |-- /* 静态页面    --> React 构建产物（直接返回）
  |-- /api/*        --> Fastify（Node.js 业务逻辑）
  |-- /ws           --> Fastify（WebSocket 升级）
  |-- /uploads/*    --> auth_request 鉴权后直接返回文件

Fastify
  |-- REST API 业务逻辑
  |-- WebSocket 实时推送
  |
  |-- PostgreSQL（业务数据）
  |-- Redis（缓存 / 队列 / Pub-Sub）
  |-- 本地文件系统（附件）
```

## 2. 前端技术栈

### 2.1 核心框架：React 19 + TypeScript

**为什么选 React 19 而非 18**：

| 对比 | React 18 | React 19 |
|------|----------|----------|
| 发布状态 | 维护模式 | 当前官方推荐稳定版（2024-12 发布） |
| Actions API | 无 | 原生支持，简化便签异步提交流程 |
| `use()` API | 无 | 统一处理 Promise / Context |
| 错误处理 | 旧 Error Boundary | 改进的错误恢复机制 |
| 主流库适配 | 完整 | Ant Design 5.x / Vite 5.x / React Router v7 均已适配 |

**结论**：直接上 React 19，无兼容风险，长期维护窗口更长。

**备选**：Vue 3（同等能力，选 React 因其更广泛的生态）

### 2.2 构建工具：Vite 5

- 开发热更新 < 100ms
- 生产构建基于 Rollup，产物体积小

### 2.3 路由：React Router v7

支持嵌套路由，适配"便签列表 → 便签详情"的多层视图结构。

### 2.4 状态管理：Zustand

- 比 Redux 轻量，API 极简
- 中间件支持持久化（离线缓存）
- 适合本项目中等复杂度状态（便签列表、当前用户、通知等）

### 2.5 CSS 样式方案：CSS Modules + CSS Variables

**为什么不用 Tailwind CSS**：
- Ant Design 已提供完整组件样式，两者混用会有优先级冲突
- Tailwind 的工具类与 Ant Design Design Token 系统职责重叠，增加维护负担

**实际方案**：

| 层次 | 方案 | 作用 |
|------|------|------|
| 组件样式 | CSS Modules（`.module.css`） | 样式天然隔离，无全局污染 |
| 全局变量 | CSS Variables（`--color-*` 等） | 统一设计 Token，深色/浅色通过切换 `:root` 变量实现 |
| 组件主题 | Ant Design `ConfigProvider.theme.token` | 定制 Ant Design 颜色/圆角等，不覆盖内部 class |

```css
/* globals.css 示例 */
:root {
  --color-bg: #ffffff;
  --color-text: #1a1a1a;
  --color-accent: #1677ff;
  --radius-card: 8px;
}
[data-theme="dark"] {
  --color-bg: #141414;
  --color-text: #e8e8e8;
  --color-accent: #4096ff;
}
```

### 2.6 UI 组件库：Ant Design 5.x

- 提供完整的表单、弹窗、通知、上传、下拉等组件，开发效率高
- 内置国际化（i18n），支持中/英切换
- 深色/浅色模式通过 `ConfigProvider` + Design Token 控制

**备选**：shadcn/ui（更轻量，但需要更多自行组装）

### 2.7 Markdown 编辑器：@uiw/react-md-editor

- 轻量，开箱即用
- 支持编辑/预览分栏模式
- 快捷工具栏可点击插入 Markdown 语法

**备选**：Milkdown（WYSIWYG 所见即所得，更现代，但包体积大）

### 2.8 日历组件：FullCalendar React

- 支持日/周/月视图切换
- 拖拽改期（`@fullcalendar/interaction`）
- 颜色自定义，循环事件原生支持（重复任务）
- 开源免费，社区版覆盖本项目所有需求

### 2.9 图表：Recharts

- 纯 React 组件，API 友好
- 支持折线图、饼图、条形图，覆盖统计模块需求
- gzip 后约 150KB，轻量

### 2.10 国际化：react-i18next

- 浏览器语言检测 + 用户设置覆盖
- 与 Ant Design i18n 配合使用
- 通知文案、界面语言、日期格式均随用户语言偏好自动切换

### 2.11 PWA：vite-plugin-pwa

- Service Worker 自动生成，离线缓存策略
- Web Push 通知基础设施

## 3. 后端技术栈

### 3.1 运行时 + 框架：Node.js 20 LTS + Fastify 4

**为什么选 Fastify 而非 Express**：

| 对比项 | Express | Fastify |
|--------|---------|---------|
| 性能（req/s） | ~15,000 | ~45,000 |
| TypeScript 支持 | 需额外配置 | 原生支持 |
| Schema 验证 | 需手动集成 | 内置 JSON Schema |
| 插件生态 | 极其丰富 | 充足，主流插件均有 |

**备选**：若团队更熟悉 Express，切换成本极低，业务逻辑无需改动。

### 3.2 语言：TypeScript（与前端共享类型定义）

前后端共用 `packages/types/` 包，减少接口对接错误。

### 3.3 ORM：Prisma 5

- 强类型，自动生成数据库客户端
- 迁移管理（`prisma migrate`）清晰可追踪
- 完整支持 PostgreSQL 所有特性

### 3.4 文件处理

- **Multer**：处理 multipart/form-data，流式写入本地磁盘
- **Sharp**：图片压缩/生成缩略图（WebP），减少存储占用

## 4. 数据库

### 核心数据库：PostgreSQL 16

- 支持 UUID、JSONB（标签数组）、原生全文搜索（`tsvector`）
- ACID 事务保障状态变更日志等关键操作一致性
- 自部署成熟，Docker 化简单

**中文全文搜索**：初版使用 `ILIKE` 模糊搜索，后期可引入 `pg_jieba` 中文分词或 Meilisearch。

### 缓存：Redis 7

- Session / JWT 黑名单存储
- 定时任务去重（防重复提醒推送）
- WebSocket 多实例消息广播（Pub/Sub）

## 5. 文件存储

### 方案：服务器本地文件系统 + Nginx 鉴权代理

**存储结构**：
```
/uploads/
  {user_id}/
    {note_id}/
      {uuid}_{原文件名}
```

**访问鉴权流程**（安全关键）：
```
客户端请求 GET /uploads/{user_id}/{note_id}/{file}
  -> Nginx 拦截，发起 auth_request 到后端 /api/auth/file
  -> 后端验证 JWT + 文件归属权限
  -> 验证通过：Nginx 用 X-Accel-Redirect 直接返回文件（不经过 Node.js）
  -> 验证失败：返回 403
```

**优点**：无外部依赖，零存储成本，性能高（Nginx 直接处理 I/O）  
**未来扩展与清理策略**：
1. **用户自助清理**：前端提供统计大屏或存储管理界面，允许用户自行判断并一键清理无用的历史废弃附件，释放磁盘空间。
2. **外部存储迁移**：存储压力过大时可无缝迁移至 MinIO（兼容 S3 API）。

## 7. 定时任务与提醒

### 任务调度：BullMQ + Redis

- **截止提醒**：创建/修改便签时写入延迟任务，到期触发 Web Push
- **重复任务生成**：cron 类型任务，按 RecurrenceRule 创建新实例
- **逾期扫描**：每日定时扫描，推送逾期通知

### Web Push：web-push（VAPID 协议）

- 标准 W3C Web Push，兼容 Chrome / Firefox / Edge / Safari（iOS 16.4+）
- 纯自部署，无第三方推送服务依赖
- **局域网环境说明**：Web Push 强制要求 HTTPS 或 `localhost`。由于系统部署在局域网，推荐实施以下方案以启用完整推送功能：
  1. **内网 HTTPS 证书分发方案（推荐）**：
     - **原理**：通过在局域网内部署自建的证书授权机构（CA），为内网服务器（如通过内部 IP 或内部域名如 `notepad.local`）签发 SSL 证书。使浏览器信任该服务器的 HTTPS 连接，从而允许注册 Service Worker 并使用 Web Push。
     - **实施步骤**：
       1. **生成根证书（Root CA）**：使用 OpenSSL 等工具生成机构专属的私有根证书。
       2. **签发服务器证书**：使用该根证书为部署服务器的内网 IP（或内部 DNS 解析的域名）签发下级 SSL 证书。
       3. **配置 Nginx**：在 Nginx 配置中加载签发的服务器证书和私钥，开启并强制 HTTPS 访问。
       4. **分发与安装根证书**：IT 部门通过活动目录（AD）组策略、移动设备管理（MDM）系统，或提供一键安装脚本，统一将 **根证书（Root CA）** 静默安装到所有员工电脑和移动设备的“受信任的根证书颁发机构”列表中。
  2. **回退方案**：若环境受限无法分发证书，系统降级为 **WebSocket 在线实时通知** + **离线应用内未读红点提示**。

## 8. 认证与安全

### JWT + Redis 黑名单

- 登录签发 **Access Token**（15分钟）+ **Refresh Token**（7天，httpOnly Cookie）
- 登出时 Refresh Token 写入 Redis 黑名单
- 密码使用 **bcrypt**（cost factor 12）哈希存储

### 文件访问：Nginx auth_request

见第 5 节，Nginx 层鉴权，Node.js 不参与文件 I/O，安全且高效。

## 9. 服务器与部署

### 部署环境：家庭局域网 (LAN)

本系统定位为**家庭版纯内部便签工具**，**服务器明确部署在本地局域网内**，无需对外网直接暴露，也无需对接复杂的企业级权限系统（如 SSO、LDAP 等），系统采用独立的独立账号密码体系（bcrypt + JWT）进行鉴权即可。局域网内的设备（电脑、手机）直接通过局域网 IP 或内部域名（如 `notepad.local`）访问各项服务。

### Nginx 与 Node.js 的职责划分

> 两者**同时存在**，职责完全不同，互为补充。

| 服务器 | 技术 | 职责 |
|--------|------|------|
| **应用服务器** | Node.js 20 + Fastify | 业务逻辑、REST API、WebSocket |
| **代理/静态服务器** | Nginx 1.25+ | HTTPS 终止、托管 React 静态文件、文件鉴权代理、转发 API 到 Fastify |

**为什么不让 Node.js 直接提供静态文件和附件**：
- Nginx 处理静态文件性能远高于 Node.js（C 实现 vs JS 事件循环）
- `auth_request` 模块可在 Nginx 层完成文件鉴权，Node.js 无需参与文件 I/O
- Nginx 天然处理 gzip 压缩、HTTP/2、TLS，无需在业务代码中实现

### 容器化：Docker + Docker Compose

```yaml
services:
  nginx:      # 反向代理 + 静态文件（挂载 /dist 和 /uploads）
  backend:    # Node.js Fastify 应用
  postgres:   # PostgreSQL 16
  redis:      # Redis 7
```

### 数据备份

- PostgreSQL：`pg_dump` 每日定时，保留 30 天
- 文件系统：`rsync` 每日增量同步到备份目录

## 11. 风险与备选方案

| 风险点 | 风险描述 | 备选方案 |
|--------|---------|---------|
| 中文全文搜索 | pg_jieba 安装复杂 | 初版 ILIKE 模糊搜索，后期引入 Meilisearch |
| Web Push 兼容性 | Safari iOS 16.4 以下不支持 | 降级为应用内 WebSocket 消息通知 |
| **局域网 Web Push 限制** | **局域网无域名或权威 HTTPS 证书导致 Service Worker / Web Push 无法注册** | **1. IT 统一部署自签发证书<br>2. 降级为在线 WebSocket 实时通知 + 离线应用内未读红点** |
| 本地文件存储扩展与冗余 | 本地硬盘空间满后影响系统运行且历史附件堆积 | 1. **提供用户自助附件清理功能**：由用户自行判断并清理<br>2. 配置磁盘告警，未来迁移至 MinIO |
| BullMQ 任务持久性 | 服务重启可能丢任务 | Redis AOF + RDB 持久化，自动恢复 |
| WebSocket 多实例 | 水平扩展时消息无法跨实例 | Redis Pub/Sub（已在方案中预设） |
