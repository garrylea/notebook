## 1. JWT 认证与 Token 生命周期管理

### 1.1 双 Token 设计方案

系统采用 **Access Token + Refresh Token** 双令牌机制，兼顾安全性与用户体验（无感刷新）。

| 令牌类型          | 存储位置                                   | 有效期      | 用途                      |
| ----------------- | ------------------------------------------ | ----------- | ------------------------- |
| **Access Token**  | 前端内存（JS 变量）                        | **15 分钟** | 每次业务 API 请求携带鉴权 |
| **Refresh Token** | `httpOnly; Secure; SameSite=Strict` Cookie | **7 天**    | 仅用于换取新 Access Token |

> **设计原因**：Access Token 存于内存（非 `localStorage`），可防止 XSS 攻击读取；Refresh Token 存于 `httpOnly Cookie`，JavaScript 无法访问，防止 XSS 窃取长效令牌。

### 1.2 无感刷新流程

```
前端 axios 请求拦截器检测到 401 (Token Expired)
    │
    ├─► 调用 POST /auth/refresh（自动携带 httpOnly Cookie 中的 Refresh Token）
    │       ├─ 成功 → 获得新 Access Token → 重放原失败请求 → 用户无感知
    │       └─ 失败 (Refresh Token 已过期 / 已被吊销)
    │               └─► 清除本地 Token → 跳转登录页
    │
    └─► 并发请求保护：刷新过程中其余 401 请求应被"挂起队列"，
        等待刷新完成后统一重放，避免并发触发多次 /auth/refresh
```

**前端实现要点**（供前端参考）：
- 使用一个全局 `isRefreshing` 标志位 + `failedQueue` 数组管理并发挂起。
- 刷新成功后遍历 `failedQueue`，`resolve` 所有挂起请求。
- 刷新失败后遍历 `failedQueue`，`reject` 所有请求并跳转登录。

### 1.3 Refresh Token 吊销机制（黑名单策略）

为支持"登出即失效"，服务端需维护一个 Refresh Token **黑名单**。

**推荐方案：Redis 黑名单**

```
用户调出 POST /auth/logout
    │
    └─► 服务端从 Cookie 中读取 Refresh Token
        ├─► 将该 Token 的 jti ( JWT ID ) 写入 Redis
        │   Key: `blacklist:jti:<jti值>`
        │   Value: 1（占位）
        │   TTL: 与该 Token 剩余有效期对齐（避免 Redis 内存无限膨胀）
        └─► 清除客户端 Cookie（Set-Cookie: refreshToken=; Max-Age=0）

用户调用 POST /auth/refresh
    └─► 服务端验证 Refresh Token 时，先查询 Redis 是否存在对应 jti
        ├─ 存在（黑名单中）→ 返回 401，拒绝刷新
        └─ 不存在 → 正常签发新 Access Token
```

> **数据库方案（备选）**：若不引入 Redis，可在 `sys_user` 表增加 `token_version`（INT）字段，每次登出 +1；JWT 中携带签发时的 version，服务端比对不一致则拒绝。该方案无 Redis 依赖，但每次鉴权需查数据库，性能较弱，适合低并发场景。

### 1.4 JWT Payload 设计规范

```json
{
  "sub": "user-uuid-xxxxx",       // 用户 ID（必填）
  "username": "lichao",           // 用户名（可选，减少查表）
  "jti": "unique-token-id-uuid",  // JWT ID（必填，用于黑名单）
  "iat": 1741046400,              // 签发时间 (issued at)
  "exp": 1741047300               // 过期时间 (expiry，iat + 15min)
}
```

> **安全约束**：JWT Payload 中**禁止存储**密码哈希、权限列表等敏感信息；签名算法使用 `HS256` 或更强的 `RS256`（推荐生产环境使用非对称 RS256，密钥对分离存储）。

## 2. 文件上传安全策略

### 2.1 文件类型白名单

服务端**不得信任**客户端传递的 MIME 类型，必须在服务端读取文件二进制魔数（Magic Bytes）进行类型验证。

| 允许分类 | 格式                             | Magic Bytes 示例                                  |
| -------- | -------------------------------- | ------------------------------------------------- |
| 图片     | `.jpg`, `.png`, `.gif`, `.webp`  | `FF D8 FF`（JPEG）, `89 50 4E 47`（PNG）          |
| 文档     | `.pdf`, `.docx`, `.xlsx`, `.txt` | `25 50 44 46`（PDF）, `50 4B 03 04`（ZIP/Office） |
| 音频     | `.mp3`, `.wav`, `.m4a`           | `49 44 33`（MP3）, `52 49 46 46`（WAV/RIFF）      |
| 视频     | `.mp4`, `.mov`                   | `00 00 00 .. 66 74 79 70`（MP4 ftyp box）         |

**被拒绝的文件类型**：`.exe`、`.sh`、`.php`、`.js`（脚本）、`.bat` 等可执行文件，直接返回 `415 Unsupported Media Type`。

### 2.2 文件大小限制

| 上传模式          | 限制     | 说明                             |
| ----------------- | -------- | -------------------------------- |
| 小文件直传        | ≤ 5 MB   | 超出则应引导前端切换为分块上传   |
| 单次分块（Chunk） | ≤ 2 MB   | 单个分片最大值，防止单块内存溢出 |
| 单文件总大小      | ≤ 100 MB | 所有分块合并后的最终限制         |

### 2.3 文件存储路径防遍历

文件落地存储路径**严禁直接使用用户传入的文件名**，必须经过以下处理：

```
用户原始文件名: "../../etc/passwd.png"
    │
    ├─► 步骤1：提取文件扩展名（仅取最后一个 . 之后的部分）
    ├─► 步骤2：生成服务端随机文件名（UUID）
    └─► 步骤3：拼接安全存储路径
             uploads/{userId}/{year-month}/{uuid}.{ext}
             例：uploads/abc123/2026-03/f47ac10b-58cc.png
```

> **绝对禁止**使用 `path.join(baseDir, userInputFilename)` 的写法，必须对路径进行规范化（`path.resolve`）并校验路径前缀是否在允许的 `baseDir` 内。

### 2.4 分块上传临时文件清理

分块上传过程中，服务端会在临时目录创建分片文件。需防范**恶意占用磁盘**的攻击：

- 每个 `upload_id` 对应的临时分片，需在 Redis/DB 中记录**最后活跃时间**。
- 定时任务（每小时执行）扫描超过 **24 小时**未完成合并的临时分片目录并删除。
- 单用户同时进行中的上传任务数量限制为 **5 个**（防止资源耗尽）。

### 2.5 附件访问鉴权（Nginx auth_request）

静态文件资源不能直接对外暴露，必须经过鉴权后才能访问：

```
客户端请求：GET /static/uploads/abc123/2026-03/file.png
    │
    └─► Nginx 收到请求，触发 auth_request 子请求
        子请求：GET /api/v1/auth/file （携带原始 Cookie / Authorization Header）
        │
        ├─ Node.js 验证 Token 有效且文件归属该用户 → 返回 200
        │   └─► Nginx 正常代理返回文件内容
        │
        └─ 验证失败 → 返回 403
            └─► Nginx 返回 403 Forbidden，拒绝文件访问
```

**Nginx 配置示意**：
```nginx
location /static/uploads/ {
    internal; # 禁止外部直接访问，仅 Nginx 内部转发可用
    auth_request /api/v1/auth/file;
    auth_request_set $auth_status $upstream_status;
    alias /data/uploads/;
}

location = /api/v1/auth/file {
    proxy_pass http://node_backend;
    proxy_pass_request_body off;
    proxy_set_header Content-Length "";
    proxy_set_header X-Original-URI $request_uri;
}
```

## 3. CORS 跨域访问策略

### 3.1 允许的源（Origins）

```
# 开发环境
Access-Control-Allow-Origin: http://localhost:5173   # Vite 开发服务器
Access-Control-Allow-Origin: http://localhost:3000

# 生产环境（严格匹配，禁止使用通配符 *）
Access-Control-Allow-Origin: https://notes.yourdomain.com
```

> **禁止在生产环境使用** `Access-Control-Allow-Origin: *`，将导致任意网站均可向 API 发送跨域请求。

### 3.2 允许的方法与请求头

```
Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization, X-Request-ID
Access-Control-Allow-Credentials: true   # 为支持 Cookie 携带，必须为 true
Access-Control-Max-Age: 86400            # 预检请求缓存 24 小时
```

### 3.3 Node.js 中间件配置示例（以 Express/cors 库为例）

```js
import cors from 'cors';

const allowedOrigins = process.env.NODE_ENV === 'production'
  ? ['https://notes.yourdomain.com']
  : ['http://localhost:5173', 'http://localhost:3000'];

app.use(cors({
  origin: (origin, callback) => {
    if (!origin || allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error(`CORS policy: origin ${origin} not allowed`));
    }
  },
  credentials: true,   // 允许携带 Cookie（Refresh Token）
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-Request-ID'],
}));
```

## 4. 全局异常处理策略

### 4.1 统一错误码规范

在标准 HTTP 状态码的基础上，扩展业务错误码（`code` 字段）以便前端精确处理。

| HTTP 状态码 | 业务 code | 场景说明                       | 前端处理建议                      |
| ----------- | --------- | ------------------------------ | --------------------------------- |
| `400`       | `40001`   | 请求参数校验失败（如标题为空） | Toast 提示具体字段错误            |
| `401`       | `40100`   | Access Token 过期或无效        | 触发无感刷新流程                  |
| `401`       | `40101`   | Refresh Token 过期 / 已吊销    | 清除 Token，跳转登录页            |
| `403`       | `40300`   | 无权限访问（如非本人便签）     | Toast: "无访问权限"               |
| `404`       | `40400`   | 资源不存在（如便签已删除）     | Toast: "内容已不存在"             |
| `409`       | `40900`   | 资源冲突（已存在同名）         | Toast: "操作冲突，请刷新"         |
| `413`       | `41300`   | 文件超过大小限制               | Toast: "文件超过 100MB 限制"      |
| `415`       | `41500`   | 不支持的文件类型               | Toast: "不支持该文件类型"         |
| `429`       | `42900`   | 请求频率超限（限流）           | Toast: "操作过于频繁，请稍后重试" |
| `500`       | `50000`   | 服务器内部错误                 | Toast: "服务异常，请稍后再试"     |

### 4.2 后端全局异常处理层

所有异常**统一在中间件层捕获并格式化**，业务代码只负责抛出，不负责输出响应。

```js
// Express 全局异常处理中间件（置于所有路由之后）
app.use((err, req, res, next) => {
  const isDev = process.env.NODE_ENV === 'development';

  // 已知业务异常（主动 throw 的 AppError）
  if (err.isAppError) {
    return res.status(err.httpStatus).json({
      code: err.code,
      message: err.message,
      error: isDev ? err.stack : undefined, // 生产环境隐藏堆栈
    });
  }

  // 参数校验失败（如 joi / zod 校验）
  if (err.isValidationError) {
    return res.status(400).json({
      code: 40001,
      message: err.details?.[0]?.message || '参数校验失败',
    });
  }

  // 未知异常兜底
  console.error('[FATAL]', err); // 写入日志系统
  return res.status(500).json({
    code: 50000,
    message: '服务器内部错误，请稍后重试',
    error: isDev ? err.message : undefined,
  });
});
```

### 4.3 接口限流策略（防暴力破解）

以下接口需配置速率限制，防止暴力破解和 DDoS 攻击：

| 接口                          | 频率限制   | 窗口时间 | 超限响应                |
| ----------------------------- | ---------- | -------- | ----------------------- |
| `POST /auth/login`            | 10 次/IP   | 15 分钟  | `429 Too Many Requests` |
| `POST /auth/register`         | 5 次/IP    | 1 小时   | `429 Too Many Requests` |
| `POST /auth/refresh`          | 30 次/IP   | 15 分钟  | `429 Too Many Requests` |
| `POST /notes/:id/attachments` | 20 次/用户 | 1 分钟   | `429 Too Many Requests` |

**推荐实现**：使用 `express-rate-limit` + Redis Store（`rate-limit-redis`）在分布式部署下共享计数器。

## 5. 其他安全加固事项

### 5.1 SQL 注入防护

- **强制使用 ORM 参数化查询**（如 Sequelize / Prisma / TypeORM），禁止拼接 SQL 字符串。
- 所有用户输入字段在写入数据库前经过 **白名单过滤**（允许的字符集），过滤 `'`、`--`、`;` 等危险字符。

### 5.2 XSS 防护

- 后端响应 Markdown 正文内容时，前端渲染前必须通过 **DOMPurify** 进行 HTML 净化。
- 响应头添加：`Content-Security-Policy: default-src 'self'` 限制脚本执行来源（生产环境按需调整）。

### 5.3 HTTPS 强制

- 生产环境 Nginx 配置 **HSTS**（`Strict-Transport-Security: max-age=31536000`），强制全站 HTTPS。
- Cookie 设置 `Secure` 属性，确保 Refresh Token 仅在 HTTPS 连接下传输。

### 5.4 敏感信息脱敏

- 日志系统（如使用的日志库）配置脱敏规则，**禁止将 `password`、`accessToken`、`refreshToken`、`Authorization` Header 字段写入日志**。
- 数据库字段中密码只存储 **Bcrypt/Argon2 哈希值**（cost factor ≥ 10），明文密码在请求处理完成后立即丢弃。
